# TASK-02: Flyway Migrations (Db2 DDL + Indexes)

Goal

Create versioned Flyway SQL migrations that define the full schema for tasks, tags, audit history, soft delete, and concurrency `version`.

Deliverables

- `src/main/resources/db/migration/V1__init.sql`
- `src/main/resources/db/migration/V2__add_indexes.sql`

Schema design (normalized tags)

- `tasks`
  - `id` (UUID string)
  - `title` required
  - `description` optional
  - `status` (`open`/`completed`)
  - `due_date` optional
  - `priority` optional
  - `created_at`, `updated_at`
  - `deleted_at` (soft delete)
  - `version` (optimistic lock)

- `tags`
  - `id` identity PK
  - `name` unique

- `task_tags` (many-to-many)
  - PK `(task_id, tag_id)`

- `task_audit`
  - records each create/update/delete/restore/hard delete
  - store before/after snapshots as JSON strings (CLOB)

Implementation steps

1) Create V1 schema (`V1__init.sql`)

Db2-oriented DDL guidance (keep it conservative):

- Use `CHAR(36)` for UUID ids.
- Use `TIMESTAMP` for time fields.
- Use `CLOB` for long text/snapshots.
- Use `GENERATED BY DEFAULT AS IDENTITY` for numeric ids.
- Add `CHECK` constraints for `status`.

Include:

- PKs and FKs.
- Unique constraint on tag name.
- Defaults for `created_at`/`updated_at` (if using DB defaults) OR keep defaults minimal and set timestamps in Java.

Recommendation (simplify cross-db tests):

- Keep DB defaults for `created_at` only.
- Set `updated_at` explicitly from Java (`@PrePersist/@PreUpdate`).

2) Create indexes (`V2__add_indexes.sql`)

Target query patterns:

- list active tasks: `deleted_at IS NULL`
- filter by status
- sort by `due_date`, `created_at` (“newest”)
- tag filtering via join
- keyword search via `LIKE` (title/description)

Recommended indexes:

- `tasks(deleted_at)`
- `tasks(status, deleted_at)`
- `tasks(due_date, deleted_at)`
- `tasks(created_at)`
- `task_tags(tag_id)` and `task_tags(task_id)`
- `tags(name)`

Text search note

Db2 full-text search is environment-specific; for now use `LIKE` in queries and an index on `title` if desired (but don’t expect it to make `LIKE '%x%'` fast).

3) Verify migrations run in tests

After TASK-01 is done, run:

- `\.\mvnw.cmd test`

If Flyway fails due to dialect differences:

- Adjust DDL to be compatible with H2 `MODE=DB2`.
- Keep Db2 correctness the priority, but ensure CI can run.

Acceptance criteria

- Flyway scripts exist and migrate cleanly on `%test` H2.
- Schema supports:
  - soft delete (`deleted_at`)
  - audit records (`task_audit`)
  - tags normalized (`tags` + `task_tags`)
  - optimistic locking (`version`)
